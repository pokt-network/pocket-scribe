# 05 â€” Multi-Version Proto Handling

## The problem

poktroll evolves. Between Shannon's alpha launch and now, the chain has gone through multiple upgrades, each potentially:
- Adding fields to protobuf types (additive, backwards-compatible).
- Renaming or removing fields (breaking).
- Changing the semantic meaning of a field (silently breaking).
- Adding new event types.
- Changing module params or behavior.

An indexer that lives for years must decode chain data from **every height since genesis** correctly â€” including heights produced by versions of the chain that no longer exist as binaries.

## The strategy

**Vendored decoders per version, dispatched by height.**

Each known poktroll version has:
- Vendored protos at `third_party/proto/poktroll/v{X}_{Y}_{Z}/`
- Generated Go types at `internal/decoders/v{X}_{Y}_{Z}/gen/`
- A `Decoder` implementation at `internal/decoders/v{X}_{Y}_{Z}/decoder.go`

The **router** (`internal/router/router.go`) picks the correct decoder for a given `block_height` based on the `upgrades` table.

## The `Decoder` interface

```go
// internal/decoders/decoder.go
package decoders

import (
    "github.com/pokt-network/pocketscribe/internal/types"
)

// Decoder converts raw chain bytes into canonical PocketScribe types.
// Each poktroll version has its own implementation.
type Decoder interface {
    // Version returns the poktroll version string (e.g. "v0.1.5").
    Version() string

    // DecodeBlock decodes a FilePlugin block file payload.
    DecodeBlock(meta, data []byte) (*types.Block, error)

    // Per-store KV decoders. Each returns NIL if the key doesn't match the store's pattern.
    DecodeSupplierKV(key, value []byte) (*types.SupplierSnapshot, error)
    DecodeApplicationKV(key, value []byte) (*types.ApplicationSnapshot, error)
    DecodeGatewayKV(key, value []byte) (*types.GatewaySnapshot, error)
    DecodeServiceKV(key, value []byte) (*types.ServiceSnapshot, error)
    DecodeSessionKV(key, value []byte) (*types.SessionSnapshot, error)
    DecodeValidatorKV(key, value []byte) (*types.ValidatorSnapshot, error)
    DecodeAccountKV(key, value []byte) (*types.AccountSnapshot, error)
    DecodeBalanceKV(key, value []byte) (*types.BalanceSnapshot, error)
    DecodeParamKV(key, value []byte) (*types.ParamSnapshot, error)

    // Per-event decoders. Return NIL if the event type doesn't match this version.
    DecodeEventClaimSettled(attrs map[string]string) (*types.EventClaimSettled, error)
    DecodeEventProofUpdated(attrs map[string]string) (*types.EventProofUpdated, error)
    DecodeEventSupplierSlashed(attrs map[string]string) (*types.EventSupplierSlashed, error)
    DecodeEventApplicationOverserviced(attrs map[string]string) (*types.EventApplicationOverserviced, error)
    DecodeEventRelayMiningDifficultyUpdated(attrs map[string]string) (*types.EventRelayMiningDifficultyUpdated, error)
    DecodeMintBurnOp(attrs map[string]string) (*types.MintBurnOp, error)
}
```

The canonical types (`internal/types/`) are **the contract** â€” they're the union of all fields that have ever existed across versions. New versions may introduce new optional fields, which means new fields on the canonical type (as pointers/nullable for backwards compat).

## The router

> **ADR-018 supersedes the "hardcoded fallback" implementation note from ADR-008. The router is DB-driven (`NewDBRouter`/`Refresh`) and version-based; there are no hardcoded heights or network branches in routing logic.**

The router lives in `internal/router/` and maps a block height to the correct decoder using the `upgrades` table as the sole source of truth (ADR-018). It is purely version-based: the only per-network input is the upgrade data (`ps sync-upgrades` populates it from the LCD); the resolution logic never branches on network identity.

**Key interfaces:**

```go
// Router is the public surface consumed by block/supplier/... handlers.
type Router interface {
    DecoderFor(height int64) (decoders.Decoder, error)
}
```

**`NewStaticRouter`** builds an in-memory snapshot from a slice of `Upgrade{Name, AppliedAtHeight, DecoderVersion}` + a `map[string]decoders.Decoder` registry + the per-network `genesisVersion` string. Used directly in unit tests (no DB required).

**`NewDBRouter`** loads the `upgrades` table once (via `store.ListUpgrades`) and builds a `staticRouter` snapshot. `Refresh` reloads the table â€” called periodically by `ps reconciler`.

**`DefaultRegistry`** (`internal/router/registry.go`) maps canonical version strings to their stateless `Decoder` implementations. Adding a new decoder version means adding one entry here.

**Lenient fallback:** unregistered intermediate versions (e.g. `v0_1_31` whose decoder ships in a later phase) fall back silently to the nearest earlier registered version. This is correct for the version-invariant block header; later phases (tx/state/event decoders) must use a strict variant that requires every version in the table to be registered.

## The upgrades table

Source of truth: poktroll's `x/upgrade` module. The indexer maintains its own `upgrades` table for fast lookup; `ps sync-upgrades` populates it from the LCD and `ps reconciler` refreshes it periodically (ADR-018).

```sql
-- in schema/migrations/0001_init.sql
CREATE TABLE upgrades (
    name                TEXT PRIMARY KEY,
    applied_at_height   BIGINT NOT NULL UNIQUE,
    applied_at_time     TIMESTAMPTZ NOT NULL,
    decoder_version     TEXT NOT NULL,
    notes               TEXT,
    indexed_at          TIMESTAMPTZ DEFAULT now()
);
```

`store.UpsertUpgrade` writes rows idempotently (`ON CONFLICT (name) DO UPDATE`). `store.ListUpgrades` returns rows ordered by `applied_at_height ASC` for the router snapshot.

## Onboarding a new version

Use `/generate-decoder vX.Y.Z`. The flow:

1. Clone `pokt-network/poktroll` at the tag.
2. Copy protos to `third_party/proto/poktroll/vX_Y_Z/`.
3. Run `buf generate` to produce Go types in `internal/decoders/vX_Y_Z/gen/`.
4. Run `buf breaking` against the previous version â†’ if breaks, write an ADR.
5. Implement `internal/decoders/vX_Y_Z/decoder.go` mapping proto types â†’ canonical types.
6. Add to the router's upgrade table.
7. Capture golden test fixtures from a node running this version.
8. Add cross-version test asserting canonical equivalence.
9. Add to CI matrix.
10. Update this doc with the new version.

Detailed steps: `.claude/agents/pocketscribe-proto-versioner.md`.

## Handling additive vs breaking changes

### Additive change: new field on an event

```proto
// v0.1.5 - EventClaimSettled added num_estimated_relays
message EventClaimSettled {
    string session_id = 1;
    string supplier_address = 2;
    string application_address = 3;
    string service_id = 4;
    cosmos.base.v1beta1.Coin settled_upokt = 5;
    int64 num_estimated_relays = 6;   // NEW in v0.1.5
}
```

Action:
- Add column to schema: `ALTER TABLE event_claim_settled ADD COLUMN num_estimated_relays BIGINT NULL` (nullable).
- New decoder (v0.1.5) populates the field; older decoders leave it `NULL`.
- Canonical type `types.EventClaimSettled` gains `NumEstimatedRelays *int64` (pointer for nullability).
- Test: same golden fixture from v0.1.0 still produces a valid `EventClaimSettled` with `NumEstimatedRelays = nil`.

### Breaking change: field renamed

```proto
// Old (v0.1.0)
message SupplierServiceConfig {
    string service_id = 1;
    Endpoint endpoint = 2;
}

// New (v0.1.5) - renamed `endpoint` to `endpoints` (now repeated)
message SupplierServiceConfig {
    string service_id = 1;
    repeated Endpoint endpoints = 2;   // was singular
}
```

Action:
- This is a wire-compatible rename in proto (same field number, same wire type for repeated singular), but semantically the canonical type must change.
- Canonical: `Services []ServiceConfig` already; the canonical `ServiceConfig.Endpoints` is `[]Endpoint`.
- Old decoder: returns a slice with 1 element (the singular endpoint).
- New decoder: returns the actual slice.
- DB column doesn't change (it's JSONB).
- Test: assert old golden â†’ `Endpoints` has 1 element.

### Semantic change: field meaning shifts

This is the **worst** kind. Example: `EventClaimSettled.settled_upokt` previously meant "gross settled before burns"; v0.2.0 redefines it as "net settled after burns".

Action:
- **Mandatory ADR with user signoff.**
- Decision options:
  - **Re-decode historical data** with the new semantic. Reindex needed.
  - **Dual-write**: keep `settled_upokt_gross` and `settled_upokt_net` columns; new decoder writes both; queries use the appropriate one based on height range.
- Default: dual-write. Less destructive.
- Test: cross-version test asserts the dual-write behaviour at the boundary.

## Cross-version testing

Every PR that touches a decoder runs the cross-version test:

```go
//go:build integration

func TestSupplierDecoder_CrossVersion_CanonicalEquivalent(t *testing.T) {
    cases := []struct {
        version  string
        fixture  string
        wantNum  int64 // expected stake_upokt
    }{
        {"v0_0_10", "v0_0_10/supplier_stake_001.bin", 1_000_000_000},
        {"v0_1_0",  "v0_1_0/supplier_stake_001.bin",  1_000_000_000},
        {"v0_1_5",  "v0_1_5/supplier_stake_001.bin",  1_000_000_000},
    }
    for _, c := range cases {
        t.Run(c.version, func(t *testing.T) {
            raw := loadFixture(t, c.fixture)
            got, err := decoders.For(c.version).DecodeSupplierKV(nil, raw)
            require.NoError(t, err)
            require.Equal(t, c.wantNum, got.StakeUpokt.Int64())
        })
    }
}
```

## Documentation discipline

When a new version is onboarded:

1. Update this doc's "Known versions" table.
2. If breaking: write `docs/decisions/ADR-NNN-poktroll-vX.Y.Z-breaks.md` documenting each break.
3. Update CI matrix.
4. Update `internal/router/upgrades.go` and the migration recording the upgrade.
