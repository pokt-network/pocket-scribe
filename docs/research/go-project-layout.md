# Go Project Layout for PocketScribe

> Researched against Grafana Tempo/Loki, HashiCorp Nomad, Celestia Node, DefiantLabs cosmos-indexer. Opinionated, decisions over options.

## 1. Survey takeaways (compressed)

| Project | Lesson |
|---|---|
| Grafana Tempo | `cmd/` (entrypoints) vs `modules/` (cohesive service units) — scales when several binaries compose the same building blocks differently. |
| Grafana Loki | Use separate in-repo Go module for things with different release cadences. |
| HashiCorp Nomad | Long-lived public Go APIs go in their own `api/` module. |
| Celestia Node | "module-per-domain with config.go/flags.go/service.go" is the most-repeated modern Cosmos-Go pattern. Adopt it. |
| DefiantLabs cosmos-indexer | **Cautionary tale** — no `internal/` boundary, everything exported, refactors break the world. Use `internal/` from day one. |

**Cross-cutting patterns**: `cmd/<binary>/main.go` thin; `internal/` for non-public; module-per-domain; codegen output committed.

### Why a single `ps` binary with subcommands (vs separate binaries)

User explicitly requested: `ps indexer`, `ps consumer supplier`, `ps fileplugin`, etc.

Benefits:
- One image to deploy, smaller surface area.
- Operators learn one CLI.
- All commands share config loading, logging, metrics, lifecycle.
- Container `ENTRYPOINT ["ps"]` with `CMD ["consumer", "supplier"]` is clean.

Tradeoff: container image is slightly larger because all subcommands' code ships. At PocketScribe's size, negligible.

## 3. CLI design (`ps` command)

```
ps                                      # show help

# RUN subcommands (long-running services)
ps fileplugin                           # the sidecar: tails FilePlugin dir → NATS
ps consumer <module>                    # run a single module consumer
ps consumer supplier
ps consumer application
ps consumer gateway
ps consumer service
ps consumer session
ps consumer tokenomics
ps consumer bank
ps consumer authz
ps indexer                              # run all enabled consumers in one process
ps reconciler                           # periodic drift detection
ps sealing                              # bucket sealing loop (can be merged into indexer)

# ADMIN subcommands (one-shot)
ps migrate up [--steps=N]               # run pending migrations
ps migrate down [--steps=N]             # rollback (rarely used)
ps migrate status                       # show migration state
ps inspect streams                      # list NATS streams + lag
ps inspect cursors                      # show consumer_consolidation
ps inspect seals [--aggregate=NAME]     # show bucket_seal status
ps replay --module=X --from=H1 --to=H2  # reindex a module range
ps backfill --from-genesis              # bulk historical load
ps reconcile --module=X [--at-height=H] # one-shot reconcile check
ps version                              # print version + build info
ps doctor                               # health check: DB conn, NATS conn, node conn
```

Naming convention:
- Run commands: present-tense action verbs or nouns (consumer, reconciler, sealing).
- Admin commands: clear verbs (migrate, inspect, replay, backfill).

## 4. Naming conventions

- `cmd/ps/main.go` (≤30 lines: build cobra root, dispatch to `internal/app/*`).
- `internal/` over `pkg/`. Only move to `pkg/` for the deliberately-public `scribeclient`.
- Package names: short, singular, lowercase, no underscores. `consumer`, not `consumers`.
- Version-suffixed packages use underscores: `v0_0_10`, not `v0010` or `v0.0.10`.
- No `util`, no `common`, no `helpers`. Domain-named instead.

## 5. Test layout

| Tier | Location | Build tag | Runner | When |
|---|---|---|---|---|
| Unit | `_test.go` next to code | none | `go test ./...` | Every commit; <10s |
| Integration | `test/integration/` | `integration` | `go test -tags=integration ./test/integration/...` | Pre-merge; testcontainers |
| E2E | `test/e2e/` | `e2e` | `go test -tags=e2e ./test/e2e/...` | Nightly + pre-release; poktroll |

Golden files: `sebdah/goldie/v2` (`-update` flag). Regenerate with `make regenerate-goldens`.

## 6. Makefile (canonical commands)

```makefile
# Build

install:                ## go install ./cmd/ps

# Codegen
gen:                    ## Run all generators
gen-proto:              ## buf generate (envelope + per-version vendored)
gen-sql:                ## sqlc generate
gen-mocks:              ## mockery
gen-check:              ## Fail if generation produces diffs (CI gate)

# Tests
test:                   ## Unit only (fast)
test-race:              ## With -race
test-integration:       ## testcontainers
test-e2e:               ## Full stack with poktroll
test-all:
test-cover:
regenerate-goldens:     ## Re-run all golden tests with -update

# Lint / format
lint:                   ## golangci-lint run
lint-fix:
fmt:                    ## gofumpt + goimports
vuln:                   ## govulncheck ./...
tidy:

# DB
db-up:                  ## docker compose up postgres
db-down:
db-migrate:             ## ps migrate up
db-reset:
db-shell:

# NATS
nats-up:
nats-streams:           ## apply stream/consumer config
nats-inspect:

# Local dev
dev:                    ## Bring up full local stack
dev-down:
run-fileplugin:
run-consumer:           ## MODULE=supplier make run-consumer
run-reconciler:

# Release
docker:
docker-push:
release:
release-snapshot:

# Meta
help:
```

Two non-negotiables:
1. `make ci` is exactly what CI runs (no drift).
2. `make gen-check` runs in CI and fails on stale codegen.

## 7. Tooling decisions

| Concern | Choice | Why |
|---|---|---|
| Linter | **golangci-lint** (pinned version) | Only choice. |
| Formatter | **gofumpt** + **goimports** | Stricter than gofmt; canonical. |
| Proto codegen | **buf** (lint + breaking + generate) | Standard. |
| SQL codegen | **sqlc** → `internal/store/gen/` | Type-safe queries with pgx v5. |
| DB driver | **pgx v5** | `database/sql` is too lossy for an indexer. |
| Mocks | **mockery v3** (YAML config) | Standard. |
| Migrations | **goose** (embed.FS) | Go-migration escape hatch matters. |
| Logging | **`log/slog`** (stdlib) | JSON in prod, text in dev. |
| Metrics | **prometheus/client_golang** | One registry per binary in `internal/metrics`. |
| Tracing | **OpenTelemetry** | Industry standard. |
| CLI | **cobra + viper** | Boring, works. |
| Build/release | **goreleaser** + **ko** for images | Multi-arch binaries; pure-Go images. |

## 8. Monorepo vs multirepo

**Single monorepo, single Go module.**

Reasons:
1. Shared types (envelope, decoder interface) are constant.
2. Atomic changes (envelope wire format = one PR).
3. One CI pipeline for integration tests.
4. Centralized codegen.
5. Monorepo cost is zero at 4 binaries.

Split later (in-repo module first, separate repo only after) when `pkg/scribeclient` gets external consumers with different cadence.

## 9. Versions to pin

| Tool | Version (latest stable as of project start) |
|---|---|
| Go | **1.24** (track minor versions deliberately) |
| PostgreSQL | **17** (latest GA; consider 18 if released by spike) |
| TimescaleDB | **2.18+** (latest OSS) |
| NATS Server | **2.10+** (latest stable) |
| NATS Go client | latest `github.com/nats-io/nats.go/jetstream` |
| pgx | **v5** latest |
| Cobra | latest |
| sqlc | latest |
| goose | latest |
| buf | latest |
| golangci-lint | latest pinned in `.golangci.yml` |

**Policy**: bump Go minor versions deliberately (each minor brings perf and stdlib upgrades). Bump major versions of deps only after testing on a feature branch.

## 10. Quick reference

If you're adding code, place it here:

| What | Where |
|---|---|
| New CLI subcommand | `internal/app/<name>/` + register in `cmd/ps/main.go` |
| New consumer module | `internal/consumer/modules/<module>/` |
| New decoder version | `internal/decoders/v<X>_<Y>_<Z>/` + `third_party/proto/poktroll/v<X>_<Y>_<Z>/` |
| New SQL query | `internal/store/queries/<entity>.sql` → run `make gen-sql` |
| New migration | `schema/<NNNN>_<description>.sql` |
| New NATS subject | Only in `internal/nats/subjects.go` |
| New metric | `internal/metrics/metrics.go` |
| New env var or config | `internal/config/<subsection>.go` |
| New aggregate | Migration in `schema/` + INSERT into `aggregate_registry` |
